Bagging Machine

Requirements

- Internals do not need to be simulated or rendered physically.
- Machine receives packs from the PCR conveyor.
- Packs should transfer from the PCR conveyor onto a first intake conveyor.
- First intake conveyor characteristics:
  - sloped in `+y` direction
  - short, but long enough to carry 2 or 3 packs
- Packs then transfer onto a second intake conveyor.
- Second intake conveyor characteristics:
  - not sloped
  - even shorter than the first intake conveyor, could carry 1 or 2 packs
- Actual intake into the machine can be a "black hole".
- Second intake conveyor can butt up against the black-hole intake.
- The black-hole intake on the bagging machine emits bags onto a connected chute.
- Intake conveyors and chute can be considered fixed relative to the machine body; positioning them independently of the machine body is not required.
- Bags slide down the chute and into a tote.
- Bag visual shape should look similar to the example in `C:\misc\img\bag.png`.
- The first bag mesh only needs the basic bag shape; decals / text are not required for the current slice.
- Tote should be considered a receiver; it is not part of the bagging machine construct.
- Tote should be on a conveyor.
- Tote should move on when full. That will be governed by a controller later, but for the test / debug rig this can initially be driven by the number of bags in the rig.
- Bagging machine should follow the same architectural principles as the existing tipping / sorting machines for:
  - building / scene placement
  - install surface and install result
  - state management and state reporting / querying

Additional clarified requirements

- The two intake conveyors and the chute are part of the bagging machine installation.
- The downstream tote conveyor remains external to the bagging machine installation.
- Bags should follow the same convention as packs for runtime ownership and rendering:
  - after bag creation they are first-class moving simulation objects
  - once received into a tote they can become children of the tote for rendering while the tote lid is open
- Packs should remain visibly moving on the two intake conveyors until they reach the black-hole intake.
- The chute should carry one completed bag at a time for the first slice.
- The bagging machine should act as the readiness boundary for the next PCR delivery:
  - intake conveyors should only run when a new group is being delivered
  - PCR and the bagger intake path should not deliver the next group until the bagging machine returns to `IDLE`
  - in practical control terms, PCR timing should be driven by bagging machine readiness rather than the bagger passively accepting whatever PCR already released

Architectural Direction

- Treat the bagging machine as an installed machine family in the same general style as the existing tipper and sorter.
- Keep machine-owned concerns out of the debug rig from the start.
- Expected ownership split:
  - `BaggingMachine`: machine-local runtime/state
  - `BaggingModule`: machine-owned renderables, intake conveyors, chute, anchor math
  - `BaggingSectionInstaller` / `BaggingInstallation`: install surface and install result
  - external tote receiver / tote conveyor: composed outside the bagging installation
- External boundaries should be explicit:
  - PCR should hand off into a bagging intake boundary
  - the chute exit should become the machine-owned downstream bag release point
- Machine internals should remain timed/stateful abstractions rather than physics-based simulation.
- The current debug-rig bagger box should be treated as temporary and replaced by a proper installed bagging module.

Recommended implementation sequence

1. First slice:
   - introduce `BaggingModule`
   - introduce `BaggingSectionInstaller` / `BaggingInstallation`
   - replace the inline bagging-machine render box in the debug rig with an installed bagging module
   - keep the current logical `BaggingMachine` runtime behavior bridged behind that new boundary
2. Second slice:
   - introduce first-class `Bag` runtime objects
   - make the bagger emit bags onto the chute / downstream receiver path
   - gate PCR release timing by bagging-machine readiness
3. Later slice:
   - add external tote receiver controller logic for full-tote move-on / replacement
   - integrate with future lid-opening / lid-closing machine work

Current code position

- The first structural bagging-machine slice is now in place.
- `BaggingModule` now exists under `warehouse.sim.totebag.assembly`.
- `BaggingSectionInstaller` / `BaggingInstallation` now exist and are used by the integrated tote-to-bag debug path.
- The integrated tote-to-bag debug installer now installs the bagger as a machine-owned assembly instead of constructing only a simple inline bagger box in the debug rig.
- The current `BaggingMachine` runtime has intentionally been left as the existing logical/timed consumer for now and is merely bridged behind the new installed-machine boundary.
- The bagging machine currently owns:
  - machine body renderable
  - first intake conveyor renderable
  - second intake conveyor renderable
  - chute renderable
  - explicit visible intake and outfeed mouth geometry
- Intake-side placement has started moving away from ad hoc local tuning:
  - first intake conveyor start is now derived from the PCR outfeed anchor with a small gap/drop policy
  - second intake conveyor start is derived from the first conveyor end
  - machine body x placement is now derived from the second conveyor end plus a small face offset
- The intake and outfeed mouth geometry is now explicitly visible on the bagging machine rather than relying only on hidden handoff points.
- The current bagger visual is intentionally still a thin-slice proving assembly:
  - acceptable for continued architecture work
  - not yet intended as final machine fidelity
- The PCR-to-bagger intake side now has an explicit readiness/reservation seam:
  - `PackGroupReceiver`
  - `PackGroupReservation`
  - controller-driven reservation before PRL release
  - bagger receive started from PCR outfeed handoff rather than by the bagger knowing concrete upstream machinery
- The PRL/PCR flow controller now depends on the `PackGroupReceiver` seam rather than on concrete `BaggingMachine`.
  - PRL release is affected by downstream readiness only through that seam.
  - Receiver fullness is hidden behind bagger/receiver availability; PRL/PCR should not inspect the bagger's downstream bag receiver directly.
- The bagger-to-receiver side now has a generic runtime-bag receiver seam:
  - `BagReceiver`
  - `BagReservation`
  - `RecordingBagReceiver`
  - `StoredBagReceiver`
  - `BaggingInstallation` exposes the installed receiver for debug/test composition
- First-class logical bags now exist:
  - `Bag`
  - logical pack contents are retained as `PackPlan` entries
  - the bag carries motion and containment state
  - bagged packs are logical contents and do not need individual renderables inside the bag
- `BaggingMachine` now creates runtime `Bag` objects from completed pack groups and passes those bags to the receiver seam.
- `BagDischarge` now exists and is wired into `BaggingMachine`:
  - it holds a `Bag`, `BagReservation`, duration, elapsed time, and progress
  - it marks the bag moving while discharge is active
  - it marks the bag received / in receiver when the discharge completes
- receiver completion is now delayed until the active discharge completes
- `BaggingMachine` now has a `WAITING_FOR_RECEIVER` state:
  - if the output receiver cannot reserve a bag after the bag has been created, the bagger waits without becoming available for another pack group
  - if the output receiver is already full while the bagger is idle, the bagger does not advertise pack-intake availability for a new group
- The debug harness now proves an external receiver path:
  - `StoredBagReceiver` stores received runtime `Bag` objects
  - the debug receiver has finite capacity
  - a debug-only auto-empty controller clears the receiver after it has been full for a short timer
  - the receiver is rendered as a simple tote-style receiver in the harness
- `BagMeshFactory` now provides a first-pass paper-bag mesh:
  - tall/narrow symbolic paper-bag shape
  - folded top and side creases
  - no decals or writing
  - still accepted as an early visual that needs later refinement

What is still left

Completed runtime/control slices

- `BagDischarge` is wired into `BaggingMachine`.
- Bag creation, discharge movement, and receiver completion are distinct lifecycle steps.
- PRL/PCR release is gated through the generic `PackGroupReceiver` seam, not through a concrete bagger reference.
- Bagger output receiver availability is handled inside the bagger / receiver boundary.
- Receiver capacity can indirectly block upstream release because a full receiver makes the bagger unavailable through the pack-group receiver seam.

Current PCR / bagger release state

- PRL release is now gated by the downstream `PackGroupReceiver` before a group is released toward PCR.
- The bagger is no longer treated as a passive concrete dependency of `ToteToBagFlowController`.
- If the bagger or its downstream bag receiver cannot reserve, upstream PRL release is blocked through the generic receiver seam.
- PRL release selection is deterministic but not strictly sequential:
  - if an earlier ready PRL cannot reserve downstream, a later ready PRL that can reserve may release
  - if multiple ready PRLs can reserve, the lowest PRL id still releases first
- PCR remains a single released-group path for the current implementation; broad PCR capacity scheduling is not the current focus.
- Remaining bagger-side control work should avoid reintroducing concrete coupling between PRL/PCR and `BaggingMachine`.

Architecture discussion to resume when bagger/PCR control is revisited

- The main open question is no longer just visual transition smoothing.
- The current receiver seam is established for this thin slice, but future adjacent-machine work should continue using small contracts rather than concrete machine references.
- Desired direction:
  - avoid concrete coupling where one machine knows named upstream/downstream machine classes
  - prefer small contracts around readiness, reservation, state query, and handoff lifecycle
  - keep machine-local state ownership inside each machine rather than having neighbours drive internal state directly
- Wider-system concern noted:
  - downstream receivers such as totes can become full
  - something external to the bagger must determine fullness and trigger hold/release or move-on behaviour
  - route/graph topology may mean there are multiple possible downstream systems rather than one fixed neighbour
- Existing transfer-zone subsystem was reviewed and should be taken seriously as its own architecture:
  - it already has state, controller, sensors, events, strategies, and mechanisms
  - it is currently route/tote-transfer oriented, not a generic mounted-machine seam abstraction
  - we should not casually "turn transfer zones into machines"
- Current recommendation:
  - keep machine architecture and transfer-zone architecture distinct for now
  - look for a shared seam vocabulary rather than forcing both into the same concrete type
  - likely shared concepts are readiness, reservation, blocked/unavailable, handoff started, and handoff completed
- This conversation should be resumed before doing major refactoring of PCR/bagger control flow.
- The current tote-to-bag priority is to tidy the bagging-machine / receiver side before moving on to the remaining warehouse machines.
- Dynamic PRL reassignment and upstream tote scheduling remain future topics after the broader warehouse layout and machine state surfaces exist.

Bagger-to-receiver seam direction

- The downstream receiver is represented as a generic runtime-bag receiver, not as a tote-specific dependency.
- The bagger should only know whether a downstream receiver can reserve and receive a bag.
- The bagger should not decide when a tote is full or when a tote should be released.
- Fullness, hold/release, and route move-on behaviour belong outside the bagger, likely in a receiver/controller/transfer-zone strategy depending on the eventual topology.
- Completed contract/domain slices:
  - `CompletedBagReceiver` / `CompletedBagReservation` were introduced and then renamed to `BagReceiver` / `BagReservation`
  - `RecordingBagReceiver` exists for debug/test composition
  - first-class logical `Bag` objects now exist
  - `BagDischarge` exists as the next transfer lifecycle primitive
  - transfer-zone classes have not been modified
