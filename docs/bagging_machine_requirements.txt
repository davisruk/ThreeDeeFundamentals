Bagging Machine

Requirements

- Internals do not need to be simulated or rendered physically.
- Machine receives packs from the PCR conveyor.
- Packs should transfer from the PCR conveyor onto a first intake conveyor.
- First intake conveyor characteristics:
  - sloped in `+y` direction
  - short, but long enough to carry 2 or 3 packs
- Packs then transfer onto a second intake conveyor.
- Second intake conveyor characteristics:
  - not sloped
  - even shorter than the first intake conveyor, could carry 1 or 2 packs
- Actual intake into the machine can be a "black hole".
- Second intake conveyor can butt up against the black-hole intake.
- The black-hole intake on the bagging machine emits bags onto a connected chute.
- Intake conveyors and chute can be considered fixed relative to the machine body; positioning them independently of the machine body is not required.
- Bags slide down the chute and into a tote.
- Bag visual shape should look similar to the example in `C:\misc\img\bag.png`.
- The first bag mesh only needs the basic bag shape; decals / text are not required for the current slice.
- Tote should be considered a receiver; it is not part of the bagging machine construct.
- Tote should be on a conveyor.
- Tote should move on when full. That will be governed by a controller later, but for the test / debug rig this can initially be driven by the number of bags in the rig.
- Bagging machine should follow the same architectural principles as the existing tipping / sorting machines for:
  - building / scene placement
  - install surface and install result
  - state management and state reporting / querying

Additional clarified requirements

- The two intake conveyors and the chute are part of the bagging machine installation.
- The downstream tote conveyor remains external to the bagging machine installation.
- Bags should follow the same convention as packs for runtime ownership and rendering:
  - after bag creation they are first-class moving simulation objects
  - once received into a tote they can become children of the tote for rendering while the tote lid is open
- Packs should remain visibly moving on the two intake conveyors until they reach the black-hole intake.
- The chute should carry one completed bag at a time for the first slice.
- The bagging machine should act as the readiness boundary for the next PCR delivery:
  - intake conveyors should only run when a new group is being delivered
  - PCR and the bagger intake path should not deliver the next group until the bagging machine returns to `IDLE`
  - in practical control terms, PCR timing should be driven by bagging machine readiness rather than the bagger passively accepting whatever PCR already released

Architectural Direction

- Treat the bagging machine as an installed machine family in the same general style as the existing tipper and sorter.
- Keep machine-owned concerns out of the debug rig from the start.
- Expected ownership split:
  - `BaggingMachine`: machine-local runtime/state
  - `BaggingModule`: machine-owned renderables, intake conveyors, chute, anchor math
  - `BaggingSectionInstaller` / `BaggingInstallation`: install surface and install result
  - external tote receiver / tote conveyor: composed outside the bagging installation
- External boundaries should be explicit:
  - PCR should hand off into a bagging intake boundary
  - the chute exit should become the machine-owned downstream bag release point
- Machine internals should remain timed/stateful abstractions rather than physics-based simulation.
- The current debug-rig bagger box should be treated as temporary and replaced by a proper installed bagging module.

Recommended implementation sequence

1. First slice:
   - introduce `BaggingModule`
   - introduce `BaggingSectionInstaller` / `BaggingInstallation`
   - replace the inline bagging-machine render box in the debug rig with an installed bagging module
   - keep the current logical `BaggingMachine` runtime behavior bridged behind that new boundary
2. Second slice:
   - introduce first-class `Bag` runtime objects
   - make the bagger emit bags onto the chute / downstream receiver path
   - gate PCR release timing by bagging-machine readiness
3. Later slice:
   - add external tote receiver controller logic for full-tote move-on / replacement
   - integrate with future lid-opening / lid-closing machine work

Architecture discussion to resume next session

- The main open question is no longer just visual transition smoothing.
- We need to settle the seam contract between adjacent systems before finalising the PCR/FIC/SIC flow.
- Desired direction:
  - avoid concrete coupling where one machine knows named upstream/downstream machine classes
  - prefer small contracts around readiness, reservation, state query, and handoff lifecycle
  - keep machine-local state ownership inside each machine rather than having neighbours drive internal state directly
- Wider-system concern noted:
  - downstream receivers such as totes can become full
  - something external to the bagger must determine fullness and trigger hold/release or move-on behaviour
  - route/graph topology may mean there are multiple possible downstream systems rather than one fixed neighbour
- Existing transfer-zone subsystem was reviewed and should be taken seriously as its own architecture:
  - it already has state, controller, sensors, events, strategies, and mechanisms
  - it is currently route/tote-transfer oriented, not a generic mounted-machine seam abstraction
  - we should not casually "turn transfer zones into machines"
- Current recommendation:
  - keep machine architecture and transfer-zone architecture distinct for now
  - look for a shared seam vocabulary rather than forcing both into the same concrete type
  - likely shared concepts are readiness, reservation, blocked/unavailable, handoff started, and handoff completed
- This conversation should be resumed before doing major refactoring of PCR/bagger control flow.

- Use fast reasoning only to implement one explicitly scoped slice at a time.

- A slice brief should state:

- Behaviour-changing slices should include or update focused tests where practical.

Bagger-to-receiver seam direction

- The downstream receiver should be represented as a generic completed-bag receiver, not as a tote-specific dependency.
- The bagger should only know whether a downstream receiver can reserve and receive a completed bag.
- The bagger should not decide when a tote is full or when a tote should be released.
- Fullness, hold/release, and route move-on behaviour belong outside the bagger, likely in a receiver/controller/transfer-zone strategy depending on the eventual topology.
- First contract-only slice:
  - add `CompletedBagReceiver`
  - add `CompletedBagReservation`
  - do not wire runtime behaviour yet
  - do not introduce first-class moving `Bag` objects yet
  - do not modify transfer-zone classes
