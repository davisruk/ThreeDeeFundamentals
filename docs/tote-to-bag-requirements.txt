.Requirements
In the warehouse there is a section that takes the contents of a tote and transfers them into bags. The tote contains packs for several different bags. The operation is conducted through several conveyers and machines.

Layout:
At the beginning is a tipping machine
A sorting machine is next to the tipping machines outflow area
Next to the sorting machine is a single span conveyer, let's call it pack delivery conveyer or PDC for short.
Perpendicular to the PDC are several single span conveyers, let's call them pack receiver links (PRL)
At the end of the PRLs is another single span conveyer (so aligned with the PDC with same dimensions), let's call it pack receiver conveyer (PCR)
At the end of the PCR is a bagging machine

Operations:
A tote arrives at the tipping machine
The tipping machine tips the tote
Packs fall from the tote into the sorting machine's receiver
The sorting machine channels the packs into single file and channels them onto the PDC
The PDC belt continually moves and carries the small packs.
On the PDC are small bumpers aligned with the PRLs
If the controller detects that a pack is destined for a PRL it instructs the bumper to push the pack onto the PRL
The PRL belt is static until told to move
The controller instructs the PRL to move forward slightly so it may receive another pack
The controller knows how many packs each PRL should contain
Once a PRL contains the correct number of packs the controller advances the belt until the first pack is near the end of the belt (close to the PCR) and then stops the belt.
The packs on each PRL are effectively now grouped and need to be delivered to the bagging machine
When the controller detects / knows the bagging machine is free it releases a PRL
The PRL belt moves and delivers all the packs it contains to the PCR and then stops again
The PCR belt is continually moving in the opposite direction to the PDC belt
The boxes travel along the PCR belt and are delivered to the bagging machine
The bagging machine drops the packs into a bag
Once all the packs are in a bag the bag is sealed and delivered to another conveyer (out of scope for now)


Analysis / Agreed Approach

General shape
This is a coordinated warehouse subsystem rather than a single machine. The primary complexity is simulation state, control flow, pack grouping, and downstream release decisions rather than rendering.

The system should be modelled at three levels:
- Physical layout and transport topology
- Warehouse simulation objects and machine state
- Controller / orchestration logic

Physical layout and transport topology
The physical layout should use conveyor-local transport rather than the broader route / segment model:
- PDC, each PRL, and PCR are modelled as local linear conveyor spans with 1D pack positions along the belt
- transfer points remain explicit and controller-owned
- initial PDC to PRL diversion can begin as a dedicated pack-diversion abstraction rather than as route-follow movement
- PRL to PCR delivery should remain a physical conveyor transfer rather than a logical teleport because grouped visible movement is part of the core behaviour

Conveyor modelling note
Future conveyor procedural generation should not inherit the same guide / opening complexity used for track transfers.
- track procedural generation currently has significant complexity around guides and openings to support transfer behaviour
- this is not required for conveyors in this tote-to-bag area
- if guides are represented for conveyors they should be treated more like simple side rails
- the conveyor should sit proud of those rails rather than being modelled as a guide-opening system
- conveyor visual / procedural work should therefore prefer simpler reusable conveyor assemblies over transfer-track-style guide logic

Warehouse simulation objects and machine state
Packs should be modelled as first-class moving simulation objects rather than as queue-only logical entities.
- each pack is a small box with dimensions
- visual size can be represented by render scaling
- packs should carry routing / grouping metadata such as correlation id
- packs should carry conveyor-local movement state because they visibly move across multiple conveyors inside this subsystem

Machine internals do not need to be physics-based.
- tipping machine should be a timed/stateful producer that emits packs from a tote manifest
- sorting machine should initially be treated as a funnel / metering stage that produces a single-file pack stream onto the PDC
- bagging machine should be a timed consumer with occupancy / free-busy state

Data model direction
The tote should carry metadata describing its contents.
- a tote manifest should define the packs expected from the tote
- each pack entry should include a correlation id linking it to a bag / PRL assignment
- order and correlation data are predetermined upstream and do not need to be inferred in the warehouse simulation

Suggested early model objects:
- ToteLoadPlan
- PackPlan
- Pack
- PrlAssignment

PRL assignment and lifecycle
Each PRL handles one bag at a time.
- packs for a correlation id are accumulated onto the assigned PRL
- the sorting machine does not guarantee pack order beyond single-file delivery onto the PDC
- PRL behaviour should therefore be explicit and stateful

Suggested PRL lifecycle:
- IDLE
- ASSIGNED
- ACCUMULATING
- READY_TO_RELEASE
- RELEASING
- IDLE

Pack motion and spacing
Packs should maintain a minimum gap on conveyors.
- the first implementation should prevent unrealistic overlap
- spacing should be treated as conveyor-flow behaviour rather than a rendering-only concern
- this behaviour is warehouse-specific and should not be forced into tote semantics

PRL indexing
On PRLs, the "move forward slightly so it may receive another pack" behaviour can initially be modelled as a fixed indexing distance rather than as sensor-driven dynamic gap control.

PCR strategy
PCR should be designed from the start as a shared downstream resource rather than as a single-bag-only conveyor.
- multiple bag groups may eventually coexist on PCR
- this implies capacity-aware release decisions
- packs therefore need dimensions and minimum-gap rules from the beginning
- conveyor occupancy should be treated as one-dimensional length along the belt, not as a 2D packing problem

Implementation guidance:
- model PCR now as stop/start capable and capacity-aware
- keep the first runtime scheduling policy conservative
- the first implementation may still release only one PRL at a time
- however, the controller and occupancy abstractions should already support downstream fit / release checks

Conveyor occupancy and release decisions
Release of a PRL onto PCR should be permissioned by controller logic rather than unconditional.
- occupancy should be based on pack lengths plus minimum gaps and safety margin
- the controller should be able to ask how much usable free downstream length exists
- release should occur only when the grouped PRL contents fit safely on PCR

Architectural direction
The design should be control-first, not geometry-first.
- define domain objects, state machines, and release rules before investing in detailed machine visuals
- use timed/stateful machine models instead of physics simulation inside enclosed equipment
- keep routing metadata and correlation decisions in controllers, not in sensors
- sensors should answer where packs are, not what the system should do with them

Recommended implementation order
1. Define the warehouse domain and control model:
   - pack dimensions
   - tote manifest / pack plan metadata
   - PRL assignment state
   - conveyor occupancy and minimum-gap rules
   - release-decision interfaces
2. Build a minimal logical / conveyor-local pack flow:
   - tipper emits packs
   - sorter meters packs
   - packs flow along PDC
   - packs are redirected to assigned PRLs
   - PRLs accumulate and index
   - PRLs release to PCR
   - bagging machine consumes released groups
3. Add conservative PCR gating using occupancy and fit checks
4. Add machine-specific mechanism behaviour and richer visuals

Key explicit decisions made so far
- Packs are real moving simulation objects, not queue-only placeholders
- Packs will carry predetermined correlation metadata derived from tote contents
- Each PRL serves exactly one bag at a time
- PDC to PRL diversion is initially instantaneous at the bumper point
- PRL indexing is initially fixed-distance
- Packs maintain a minimum gap on conveyors
- Pack dimensions matter from the beginning because PCR fit / release logic depends on them
- PCR should be architected now as a shared, capacity-aware downstream conveyor
- Conveyor occupancy should be modelled as 1D length along the conveyor

Implemented so far

Completed domain and control model
The tote-to-bag domain and first control/state-machine slice now exist in code.
- tote load planning is represented by `ToteLoadPlan`
- pack planning and pack runtime objects exist
- PRL assignment, lifecycle, and release decisions exist
- PCR occupancy and release-fit checks exist
- machine states for tipping, sorting, and bagging exist
- bagging output is now modelled as a fixed-size configurable bag via `BagSpec`

Completed logical flow
A minimal end-to-end logical tote-to-bag flow now works and is covered by a test.
- tipper emits packs from a tote load plan
- sorter meters packs into single-file flow
- packs are assigned to PRLs by correlation id
- PRLs accumulate and then release when PCR capacity allows
- PCR delivers groups to the bagging machine
- the bagging machine consumes one group into one bag
- a JUnit smoke test currently verifies this thin-slice flow

Completed in-app debug harness
A runnable tote-to-bag debug harness now exists in the active scene wiring.
- machines, conveyors, packs, and completed bags are visible in-app
- selection inspection is wired into the debug panel for the tote-to-bag rig objects
- the harness currently uses simple block visuals for machines, packs, and bags
- the harness has been sufficient to prove the current state-machine and controller shape

Completed conveyor visual direction
The temporary placeholder conveyor boxes have been replaced in the harness.
- PDC, PRLs, and PCR now use reusable `StraightConveyorFactory` assemblies
- the tote-to-bag direction is now aligned with the broader conveyor direction already established elsewhere in the codebase
- this confirms the preferred modelling direction: reusable straight conveyor assemblies rather than transfer-track-style procedural conveyor generation

Completed stop / start conveyor visuals
The harness now has runtime-controlled conveyor visuals.
- a conveyor runtime state layer now allows visual start / stop control
- PDC currently runs continuously
- PRLs visually run when they hold packs or are ready to release
- PCR visually runs when it contains or is delivering work
- this proves the stop / start visual mechanism needed for later fuller implementation

Completed initial PDC transfer capability
The harness now has a first controller-owned PDC diversion stage.
- sorter output no longer jumps directly into a PRL
- packs now spend time in a PDC transfer stage before arriving at the target PRL
- a bumper renderable is mounted per PRL on the PDC
- the active bumper is driven from live transfer state
- this currently proves controller ownership and visual routing intent, not final physical fidelity

Completed geometry refinement for the harness
The tote-to-bag harness geometry has been tightened enough to validate direction.
- conveyors are now much narrower and read more like single-pack conveyors
- PDC bumpers have been moved to the opposite side of the conveyor to better match the intended real arrangement
- PDC transfer motion now reads as travel along the conveyor followed by a side diversion into the target PRL rather than a single rough diagonal drift
- PRL and PCR pack placement is now closer to conveyor centerlines rather than broad placeholder offsets

Completed conveyor-local pack transport slice
The harness is no longer relying only on grouped placeholder positioning for PRLs and PCR.
- `LinearConveyorLane` and `LinearLaneEntrySnapshot` now model 1D pack positions along a conveyor span inside `warehouse.sim.totebag`
- PRLs now own lane-backed pack positions rather than only a pack list plus indexed-distance bookkeeping
- PCR now receives groups incrementally as packs arrive from a releasing PRL rather than only by whole-group timer completion
- PCR entry is now no longer forced back to the start of the PCR span:
  released PRL packs can join the PCR at a PRL-specific entry distance and continue downstream toward the bagger from that join point
- the debug harness now renders PRL and PCR pack positions from simulation lane state rather than broad grouped offsets
- a focused JUnit flow test still passes after the movement changes

Completed PRL control refinement
PRL behaviour now more closely matches the intended warehouse sequencing.
- on partial fill, a PRL indexes forward by a fixed distance and then stops
- once a PRL has all assigned packs, it stages the grouped packs toward the PCR end and then stops in a waiting state
- PRLs no longer auto-release just because they are full
- the controller now allows only one PRL release at a time and does not start a new release while PCR or the bagger still has work in flight

Completed harness timing refinement
The tote-to-bag debug harness has been retuned to make the movement behaviour easier to inspect.
- PDC transfer timing is now derived from a fixed transfer speed so all PRL destinations read at a more uniform apparent speed
- the demo tote pack order has been mixed so PRL destinations are less obviously sequential in the visual test
- PRL belt speed is now intentionally much faster than PDC and PCR to better match the intended indexing behaviour
- fallback PDC ghost-pack placement was removed from the harness after verification that it was a render bug rather than the intended flow

Completed explicit PRL to PCR handoff slice
The release side is now materially closer to the intended conveyor effect.
- PRL to PCR transfer is now an explicit simulation-owned handoff state rather than an implicit immediate boundary hop
- the debug harness now renders the PRL to PCR handoff path explicitly
- transferred packs now join the PCR at the correct downstream join point for their PRL rather than being inserted at the PCR start
- the earlier per-PRL handoff throttling was removed after validation because it introduced an unwanted pause between packs
- the harness PCR speed is now aligned with PRL release speed so grouped release reads as continuous rather than stop-start
- the current result is much closer to the intended visible effect even though the handoff path itself is still harness-owned geometry

Current status
The current tote-to-bag implementation should be treated as a proved thin slice rather than as finished production behaviour.
- controller/state-machine direction is now materially validated
- conveyor visual direction is also materially validated
- the current scene is still a debug harness and not yet the final warehouse subsystem implementation
- pack motion is now materially closer to the intended conveyor-local model on PRLs and PCR
- PRL to PCR release now has the intended continuous downstream effect in the harness
- the main remaining transport approximation is now on the PDC side rather than on the PRL release side

What remains

Pack movement fidelity
- replace the remaining harness-style movement on the PDC side with the same conveyor-local transport direction now used for PRLs and PCR
- keep improving transfer motion so packs stay simulation-owned throughout handoff rather than dropping between staged visual states
- only consider generalising the linear transport model after a second real use case appears outside tote-to-bag

Transfer fidelity
- make PDC to PRL transfer read more like a true side-push/divert mechanism rather than the current simplified staged interpolation
- treat PDC to PRL and PRL to PCR as distinct mechanism shapes unless a later second use case proves a shared abstraction is worthwhile
- add the real transfer-device runtime state and controller boundary rather than just the current bumper visual actuation

Conveyor ownership and behaviour
- move the remaining runtime ownership on the PDC side out of the harness and into the tote-to-bag subsystem/controller layer
- only revisit PCR release/startup timing if later scheduling work shows a real downstream-control need; the current continuous release effect is now the preferred visual behaviour
- decide whether PCR should remain continuously running in the intended behaviour or become more explicitly command-driven

Machine fidelity
- tipping, sorting, and bagging remain block-like placeholders visually
- if improved visuals are needed later they should be added after behaviour ownership is stable
- machine internals should continue to be timed/stateful abstractions rather than physics simulations

Integration work
- decide how the tote-to-bag subsystem should live alongside or inside the broader warehouse scene setup
- decide whether the tote-to-bag flow becomes a dedicated example scene, a reusable warehouse module, or a subsection embedded in a larger layout
- propagate inspection support to any new tote-to-bag runtime objects introduced in later phases

Current recommended next step
- continue from the proved harness toward actual subsystem integration by finishing the PDC side of the movement model:
  - move PDC-side movement ownership further into the conveyor-local transport layer so packs remain simulation-owned from sorter output through PRL arrival
  - replace the current timed PDC diversion approximation with a more explicit side-push / divert handoff model owned by the subsystem rather than by harness-only path math
  - keep the current PRL to PCR behaviour direction unless later scheduling or layout work reveals a concrete problem with the continuous release effect
  - keep machine cosmetics secondary until the remaining transport ownership and handoff behaviours are stable

Branching strategy
This work is expected to span multiple sessions and should not be forced into a single long-running unstructured branch.

Recommended branch shape
- keep `master` as the stable integration branch
- treat `conveyer` as the current long-lived parent branch for ongoing conveyor / warehouse transport direction work
- create short-lived child branches from `conveyer` for tote-to-bag increments
- merge completed tote-to-bag child branches back into `conveyer`
- only merge to `master` when the tote-to-bag work reaches a coherent, demonstrable milestone

Why this fits the current repo
- current active development is already on `conveyer`
- tote-to-bag depends on conveyor modelling choices and should stay close to that evolving work
- isolating each increment in a child branch keeps partial work reviewable and reduces session-to-session risk
- `master` should not absorb half-finished warehouse flow mechanics

Recommended naming
Use focused branches for individual slices of work, for example:
- `feature/tote-to-bag-domain-model`
- `feature/tote-to-bag-pack-flow`
- `feature/tote-to-bag-prl-control`
- `feature/tote-to-bag-pcr-gating`
- `feature/tote-to-bag-machines`

Recommended increment boundaries
Prefer branches that land one coherent vertical slice at a time:
- domain model and pack metadata
- pack object plus basic route-backed movement
- PRL assignment / indexing / accumulation
- PCR occupancy and conservative release gating
- machine-specific behaviour and visuals

Working rule for future sessions
- start each session from the latest `conveyer`
- create a fresh tote-to-bag feature branch for the specific slice being worked on
- avoid mixing unrelated conveyor refactors into the same tote-to-bag branch unless they are required prerequisites
- if prerequisite conveyor refactor work is needed, land that separately into `conveyer` first and then rebase or branch again for tote-to-bag work

Merge policy
- merge feature branches into `conveyer` only when they compile and leave the simulation in a runnable state
- prefer several small merges over one very large branch covering the whole tote-to-bag area
- defer merge to `master` until the subsystem has at least a stable end-to-end thin slice
