.Requirements
In the warehouse there is a section that takes the contents of a tote and transfers them into bags. The tote contains packs for several different bags. The operation is conducted through several conveyers and machines.

Layout:
At the beginning is a tipping machine
A sorting machine is next to the tipping machines outflow area
Next to the sorting machine is a single span conveyer, let's call it pack delivery conveyer or PDC for short.
Perpendicular to the PDC are several single span conveyers, let's call them pack receiver links (PRL)
At the end of the PRLs is another single span conveyer (so aligned with the PDC with same dimensions), let's call it pack receiver conveyer (PCR)
At the end of the PCR is a bagging machine

Operations:
A tote arrives at the tipping machine
The tipping machine tips the tote
Packs fall from the tote into the sorting machine's receiver
The sorting machine channels the packs into single file and channels them onto the PDC
The PDC belt continually moves and carries the small packs.
On the PDC are small bumpers aligned with the PRLs
If the controller detects that a pack is destined for a PRL it instructs the bumper to push the pack onto the PRL
The PRL belt is static until told to move
The controller instructs the PRL to move forward slightly so it may receive another pack
The controller knows how many packs each PRL should contain
Once a PRL contains the correct number of packs the controller advances the belt until the first pack is near the end of the belt (close to the PCR) and then stops the belt.
The packs on each PRL are effectively now grouped and need to be delivered to the bagging machine
When the controller detects / knows the bagging machine is free it releases a PRL
The PRL belt moves and delivers all the packs it contains to the PCR and then stops again
The PCR belt is continually moving in the opposite direction to the PDC belt
The boxes travel along the PCR belt and are delivered to the bagging machine
The bagging machine drops the packs into a bag
Once all the packs are in a bag the bag is sealed and delivered to another conveyer (out of scope for now)


Analysis / Agreed Approach

General shape
This is a coordinated warehouse subsystem rather than a single machine. The primary complexity is simulation state, control flow, pack grouping, and downstream release decisions rather than rendering.

The system should be modelled at three levels:
- Physical layout and transport topology
- Warehouse simulation objects and machine state
- Controller / orchestration logic

Physical layout and transport topology
The physical layout should use conveyor-local transport rather than the broader route / segment model:
- PDC, each PRL, and PCR are modelled as local linear conveyor spans with 1D pack positions along the belt
- transfer points remain explicit and controller-owned
- initial PDC to PRL diversion can begin as a dedicated pack-diversion abstraction rather than as route-follow movement
- PRL to PCR delivery should remain a physical conveyor transfer rather than a logical teleport because grouped visible movement is part of the core behaviour

Conveyor modelling note
Future conveyor procedural generation should not inherit the same guide / opening complexity used for track transfers.
- track procedural generation currently has significant complexity around guides and openings to support transfer behaviour
- this is not required for conveyors in this tote-to-bag area
- if guides are represented for conveyors they should be treated more like simple side rails
- the conveyor should sit proud of those rails rather than being modelled as a guide-opening system
- conveyor visual / procedural work should therefore prefer simpler reusable conveyor assemblies over transfer-track-style guide logic

Warehouse simulation objects and machine state
Packs should be modelled as first-class moving simulation objects rather than as queue-only logical entities.
- each pack is a small box with dimensions
- visual size can be represented by render scaling
- packs should carry routing / grouping metadata such as correlation id
- packs should carry conveyor-local movement state because they visibly move across multiple conveyors inside this subsystem

Machine internals do not need to be physics-based.
- tipping machine should be a timed/stateful producer that emits packs from a tote manifest
- sorting machine should initially be treated as a funnel / metering stage that produces a single-file pack stream onto the PDC
- bagging machine should be a timed consumer with occupancy / free-busy state

Data model direction
The tote should carry metadata describing its contents.
- a tote manifest should define the packs expected from the tote
- each pack entry should include a correlation id linking it to a bag / PRL assignment
- order and correlation data are predetermined upstream and do not need to be inferred in the warehouse simulation

Suggested early model objects:
- ToteLoadPlan
- PackPlan
- Pack
- PrlAssignment

PRL assignment and lifecycle
Each PRL handles one bag at a time.
- packs for a correlation id are accumulated onto the assigned PRL
- the sorting machine does not guarantee pack order beyond single-file delivery onto the PDC
- PRL behaviour should therefore be explicit and stateful

Suggested PRL lifecycle:
- IDLE
- ASSIGNED
- ACCUMULATING
- READY_TO_RELEASE
- RELEASING
- IDLE

Pack motion and spacing
Packs should maintain a minimum gap on conveyors.
- the first implementation should prevent unrealistic overlap
- spacing should be treated as conveyor-flow behaviour rather than a rendering-only concern
- this behaviour is warehouse-specific and should not be forced into tote semantics

PRL indexing
On PRLs, the "move forward slightly so it may receive another pack" behaviour can initially be modelled as a fixed indexing distance rather than as sensor-driven dynamic gap control.

PCR strategy
PCR should be designed from the start as a shared downstream resource rather than as a single-bag-only conveyor.
- multiple bag groups may eventually coexist on PCR
- this implies capacity-aware release decisions
- packs therefore need dimensions and minimum-gap rules from the beginning
- conveyor occupancy should be treated as one-dimensional length along the belt, not as a 2D packing problem

Implementation guidance:
- model PCR now as stop/start capable and capacity-aware
- keep the first runtime scheduling policy conservative
- the first implementation may still release only one PRL at a time
- however, the controller and occupancy abstractions should already support downstream fit / release checks

Conveyor occupancy and release decisions
Release of a PRL onto PCR should be permissioned by controller logic rather than unconditional.
- occupancy should be based on pack lengths plus minimum gaps and safety margin
- the controller should be able to ask how much usable free downstream length exists
- release should occur only when the grouped PRL contents fit safely on PCR

Architectural direction
The design should be control-first, not geometry-first.
- define domain objects, state machines, and release rules before investing in detailed machine visuals
- use timed/stateful machine models instead of physics simulation inside enclosed equipment
- keep routing metadata and correlation decisions in controllers, not in sensors
- sensors should answer where packs are, not what the system should do with them

Recommended implementation order
1. Define the warehouse domain and control model:
   - pack dimensions
   - tote manifest / pack plan metadata
   - PRL assignment state
   - conveyor occupancy and minimum-gap rules
   - release-decision interfaces
2. Build a minimal logical / conveyor-local pack flow:
   - tipper emits packs
   - sorter meters packs
   - packs flow along PDC
   - packs are redirected to assigned PRLs
   - PRLs accumulate and index
   - PRLs release to PCR
   - bagging machine consumes released groups
3. Add conservative PCR gating using occupancy and fit checks
4. Add machine-specific mechanism behaviour and richer visuals

Key explicit decisions made so far
- Packs are real moving simulation objects, not queue-only placeholders
- Packs will carry predetermined correlation metadata derived from tote contents
- Each PRL serves exactly one bag at a time
- A PRL / bag assignment can span multiple source totes:
  packs from tote A and tote B may both be destined for the same PRL and eventual bag
  tote boundaries must therefore not reset PRL assignment state
- PDC to PRL diversion is initially instantaneous at the bumper point
- PRL indexing is initially fixed-distance
- Packs maintain a minimum gap on conveyors
- Pack dimensions matter from the beginning because PCR fit / release logic depends on them
- PCR should be architected now as a shared, capacity-aware downstream conveyor
- Conveyor occupancy should be modelled as 1D length along the conveyor
- The tipper should not own tote-injection policy:
  an upstream tote injector should observe tipper readiness and feed totes when the tipper can accept them
- The tipper should gate local tipping/discharge on downstream readiness rather than on knowledge of the full tote-to-bag plan

Cross-tote assignment direction
- `ToteToBagBatchPlan` now represents expected total pack counts per bag correlation separately from individual tote manifests.
- Each tote manifest contributes some subset of packs to those batch-level correlations.
- `ToteToBagBatchPlan.fromToteLoadPlans(...)` aggregates expected counts across multiple tote manifests.
- PRLs are assigned to active bag correlations, not to a tote.
- A PRL remains assigned across tote boundaries until the expected pack count for that bag is reached.
- Tote boundaries do not clear or recreate PRL assignments.
- Once the PRL releases its completed group, it is eligible for reassignment to another incomplete bag correlation.
- Dynamic PRL reassignment is arrival-driven:
  - initial assignment seeds only as many correlations as there are PRLs
  - when a pack arrives for an unassigned batch correlation, the lowest-id idle PRL is assigned to that correlation
  - if no PRL is idle, the controller reports a local admission failure; upstream scheduling must avoid presenting such a tote
- Correlation ids should remain globally unique or namespaced enough to avoid collisions when multiple totes are fed through the same run.

Tote injector / tipper responsibility split
- A debug tote injector now owns queued source totes for the integrated harness.
- The injector observes whether the tipper can accept a tote through `ToteTrackTipperFlowController.canAcceptNextTote()`.
- The injector does not decide when the tote is tipped or how packs are discharged.
- The tipper owns local capture, tip, discharge, and release behaviour.
- The tipper only starts tipping when its downstream flow can accept discharged packs.
- If the sorter/PDC/PRL side is blocked, the tipper should hold the tote rather than forcing packs downstream.
- Because the tipper cannot control which packs fall from the tote, downstream readiness must include a full-tote admission check before tipping:
  - every correlation in the candidate tote must already have an active PRL assignment or be able to obtain an idle PRL
  - the check must count distinct new correlations, not individual packs
  - if the full tote cannot be admitted, the tipper should hold the tote until enough PRLs have released and become idle
  - this is a local tote-to-bag cell readiness rule, not global tote scheduling
- This mirrors the existing downstream-readiness/backpressure approach used elsewhere:
  components expose readiness/reservation seams and upstream systems respond indirectly.

Deferred scheduler work
- Dynamic PRL reassignment after completed release is now implemented locally in `ToteToBagFlowController`.
- The controller responsibility is local:
  preserve active assignments, release completed groups, and reuse idle PRLs when packs for unassigned batch correlations arrive.
- The controller should not decide tote order.
- A slim deadlock remains possible if every PRL is reserved for an incomplete bag and the next tote contains only packs for new, unassigned bags.
- Tote sequence must therefore be guaranteed by a future scheduler.
- The scheduler should choose a tote that can advance currently assigned correlations or can introduce new correlations only when there is idle PRL capacity.
- Before the full scheduler exists, the tote-to-bag cell now uses a local admission gate so the tipper does not tip a tote whose complete contents cannot currently be accepted.
- The earlier visual 15-PRL / 40-pack failure `No idle PRL available for correlation bag-r` is the motivating case for that local admission rule.
- A future tote-to-bag cell API may expose a scheduler-facing query such as whether a candidate tote can make progress, but the scheduling decision should remain outside the tipper and outside `ToteToBagFlowController`.
- Scheduler requirements are expected to be large because they depend on tote release rules, buffer state, routing constraints, machine availability, and the state of the broader warehouse floor.
- Do not make scheduler API design the next implementation slice unless a concrete machine/layout slice forces the issue.

Deferred PCR / exception handling considerations
- The PCR could eventually become bag-aware and carry more than one released bag group while the bagger is busy.
- That is intentionally not the next fix because it introduces harder rules:
  - PRL release would need to verify enough PCR space between bag groups for the entire PRL contents
  - PCR would need explicit group-boundary spacing and delivery ordering
  - bagger reservation timing would need to cover multiple waiting groups
- The current conservative PCR policy remains one released group in flight while broader warehouse control is still forming.
- Future ray-casting / debug command buttons may allow manual intervention when a deadlock or exception state occurs.
- A manual release should mark the affected bag or flow as an exception rather than silently treating it as normal output.
- The receiving tote would then need to visit a future exception station.
- Exception handling and command buttons should be designed as part of the broader debug/control tooling, not folded into the immediate PRL admission fix.

Route-mounted machine alignment note
The current isolated tipper work has now clarified an architectural point that should guide later integration work:
- the tipper should follow the same general integration conventions as mounted transfer machines
- however, it should not be forced into transfer-zone-specific semantics when its responsibilities differ materially

Shared convention with transfer machines
The tipper should align with transfer machines in these ways:
- machine placement should be spec/layout-driven rather than ad hoc scene wiring
- the machine should be mounted against route/track infrastructure rather than treated as a free-floating scene object
- the machine should be assembled as a package of runtime/controller pieces plus renderable assembly
- the tote interaction boundary should be explicit and machine-owned rather than hidden inside harness-only checks
- scene-level wiring should install/register the machine rather than constructing all internals inline

Why the tipper is not identical to a transfer machine
The tipper differs from the current steering/transfer devices in three important ways:
- it behaves more like a specialised track section that the tote rides onto and from, not only a side-mounted branch mechanism
- it captures, holds, visually tilts, and later releases a tote, which is a stronger interaction with route-follow movement than current transfer readiness/branching logic
- it is the explicit boundary where tote route transport transitions into pack conveyor-local transport

Agreed direction
The preferred direction is therefore:
- define and follow a broader route-mounted machine convention
- treat current transfer machines as one route-mounted-machine family
- treat the tipper as another route-mounted-machine family
- keep shared conventions aligned even where concrete topology/runtime objects differ

What should be shared
The following should stay aligned between transfer machines and the eventual reusable tipper module:
- spec-driven placement
- machine assembly/builder shape
- controller/runtime/renderable split
- route-scene installation/registration conventions
- explicit tote/machine interaction boundary
- attachment-anchor exposure for downstream equipment

What may remain different
The following do not need to be forced into a single identical abstraction immediately:
- transfer-zone topology objects
- trigger/readiness semantics
- mechanism interfaces
- the exact runtime state machine for tote interaction

Follow-on slice: tote, bag, and pack rendering policy
The rendering relationship between totes, bags, and packs should now be treated as an explicit follow-on requirement rather than as an incidental harness detail.

Tote-contained pack rendering
- packs inside a tote should remain individually renderable because the tote interior is visible in the tipper slice
- contained-pack visuals should use a simple deterministic tote-local layout rather than overlapping in the same space
- the first implementation should use tote inner dimensions plus pack outer dimensions to place packs in evenly spaced rows across the tote floor
- once the current floor layer is full, additional packs should start a new layer above it
- all contained packs may initially share the same yaw/orientation; no physics, collision solving, slope solving, or contact-angle modelling is required
- this layout requirement is visual-placement logic only and should not be treated as a general rigid-body capability

Free pack rendering
- once a pack leaves tote containment and becomes visible in discharge or conveyor-local transport, it should continue to render as its own top-level renderable object
- current runtime attach/detach behaviour between container-local rendering and top-level scene rendering is an acceptable direction for this subsystem

Bag-contained pack rendering
- once packs are inside a bag, the bag interior is not expected to be rendered
- bagged packs should remain part of the simulation/domain model so bag contents, counts, and downstream logic stay correct
- however, bagged packs do not need to remain individually visible renderables
- it is acceptable for bagged packs to be removed from active rendering or otherwise excluded from the render pipeline while still remaining in simulation state

Rendering policy summary
- tote containment: packs are visible and laid out in tote-local space
- transfer/conveyor motion: packs are visible as free top-level renderables
- bag containment: packs remain simulation-owned but are not expected to remain individually visible
